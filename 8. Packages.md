Go code is organized into **packages**, which are grouped into **modules**. This hierarchical structure is essential for code reuse, visibility control, and dependency management.

### 8.0 Overview: Structure of a Go Project

|**Component**|**Description**|**Role**|**Tooling**|
|---|---|---|---|
|**Package** (e.g., `fmt`)|A directory of source files with a common purpose.|The unit of **code reuse** and **visibility** (exported/unexported identifiers).|`import` keyword|
|**Module** (e.g., `example.com/project`)|A collection of related packages.|The unit of **versioning** and **dependency management**.|`go.mod`, `go mod tidy`|

### Key Concepts

- **Visibility Control:** Whether an identifier (function, variable, etc.) is **exported** (public) or **unexported** (private) is determined solely by its capitalization.
    
- **Module Path:** Defined in the `go.mod` file, this is the canonical name used to import the module's packages.
    
- **Entry Point:** An executable program must reside in the special **`package main`** and contain a **`func main()`**.
